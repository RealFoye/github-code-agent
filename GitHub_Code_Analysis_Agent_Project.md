> **岗位方向**：Agent 应用工程师 / Agent 开发工程师（通用）
> **项目难度**：⭐⭐⭐�☆（中级偏上）
> **建议周期**：3-4 周
> **适合人群**：计算机相关专业学生、转行者，有 Python 基础即可起步

---

## 一、项目概述

### 这是什么？

一个**智能 GitHub 项目发现与深度分析 Agent**。用户只需用自然语言描述自己想找什么类型的项目，Agent 会自动在 GitHub 上搜索、筛选、对比，帮用户锁定最合适的项目，然后对选定项目进行代码级深度分析。

### 核心价值

> 模拟了真实 AI 工程师的工作场景——**从信息检索到深度理解到辅助决策**，这是企业里最需要的 Agent 能力闭环。

### 用户使用流程

```
用户输入需求："我想找一个轻量级的 Python Web 框架项目学习"
        ↓
   ┌──── Agent 第一阶段：智能搜索与推荐 ────┐
   │                                           │
   │  1. 理解用户意图（关键词提取 + 分类）       │
   │  2. 调用 GitHub API / Serper 搜索           │
   │  3. 从搜索结果中筛选 Top N 个候选仓库        │
   │  4. 对每个候选做初步评估（Stars/Forks/活跃度）│
   │  5. 输出结构化推荐卡片给用户选择              │
   │                                           │
   └───────────────────────────────────────────┘
        ↓
  用户挑选感兴趣的项目（可多选）
        ↓
   ┌──── Agent 第二阶段：深度代码分析 ──────────┐
   │                                           │
   │  6. 克隆选中的仓库                          │
   │  7. 解析代码结构（AST 分析）                 │
   │  8. 构建代码级 RAG 索引                     │
   │  9. 支持自然语言提问和回答                   │
   │  10. 生成项目概览报告                       │
   │                                           │
   └───────────────────────────────────────────┘
```

## 二、功能模块详解

### 🎯 阶段一：智能搜索与项目推荐

#### 1.1 需求理解模块

| 功能 | 描述 | 实现方式 |
|------|------|---------|
| 意图识别 | 判断用户要找什么类型的项目 | LLM 分类（框架/工具/库/教程/Demo） |
| 关键词提取 | 从自然语言中提取搜索关键词 | Prompt Engineering 或 NER |
| 技术栈推断 | 推断用户可能倾向的技术栈 | LLM 推理 + 常见映射表 |

**示例对话：**
```
用户：我想找一个适合新手学习的全栈项目，最好是用 React 和 Node.js 的
Agent：我理解你想找：
  - 类型：全栈应用（Full-stack Application）
  - 技术栈：React + Node.js
  - 难度：入门友好
  - 目的：学习参考
  
  正在为你搜索匹配的 GitHub 项目...
```

#### 1.2 项目搜索模块

| 功能 | 描述 | 实现方式 |
|------|------|---------|
| GitHub API 搜索 | 调用 GitHub Search API | PyGithub 或直接 HTTP |
| 补充搜索引擎 | 用 Google/Serper 扩展搜索 | Serper.dev API（有免费额度） |
| 多维度筛选 | 按 Stars、语言、更新时间、License | 组合过滤条件 |
| 结果去重 | 同一仓库不同搜索结果合并 | 仓库 URL 归一化 |

**关键点：** 不能只返回原始搜索结果，要做**相关性排序**——用 LLM 对每个结果打分（与用户需求的匹配度），而不是只看 Stars 数。

#### 1.3 项目评估与推荐

每个候选仓库需要生成一张**结构化推荐卡片**：

```json
{
  "repo_name": "fullstack-starter-kit",
  "author": "awesome-dev",
  "stars": 4200,
  "forks": 890,
  "language": ["TypeScript", "Python"],
  "last_updated": "2025-12-15",
  "open_issues": 12,
  "license": "MIT",
  "description": "A production-ready fullstack starter template...",
  "relevance_score": 0.92,
  "suitability_reasons": [
    "包含完整的 React + Node.js 技术栈",
    "README 详细，适合新人上手",
    "最近仍在积极维护"
  ],
  "potential_concerns": [
    "TypeScript 占比较高，需要一定基础"
  ],
  "tags": ["fullstack", "starter", "react", "nodejs", "beginner-friendly"]
}
```

**输出形式：** 终端表格 / Web 界面卡片列表，用户可以输入编号选择

#### 1.4 用户交互选择

- 支持多选（用户可以选择 1-3 个项目进行深入分析）
- 支持"都不满意"→重新搜索（调整关键词或扩大范围）
- 选择确认后进入第二阶段

### 🔍 阶段二：深度代码分析

#### 2.1 代码仓库解析

| 功能 | 描述 | 实现方式 |
|------|------|---------|
| 仓库克隆 | 克隆到本地临时目录 | GitPython 或 subprocess |
| 目录结构扫描 | 识别项目骨架 | 递归遍历 + 忽略规则处理 |
| 文件分类 | 区分源码/配置/文档/测试 | 后缀名 + 特殊文件名 |
| AST 解析 | 提取函数/类/导入关系 | Tree-sitter（推荐）或 ast 模块 |
| 依赖分析 | 读取 package.json/requirements.txt | 解析依赖声明文件 |

#### 2.2 代码级 RAG 构建

**这是整个项目的核心工程难点：**

```
源代码文件
    ↓
[切片策略] ← 不是按字符数切！而是按语义单位切
    ↓
┌─ 函数级切片：每个函数/方法作为一个 Chunk
├─ 类级切片：类定义作为一个 Chunk
├─ 模块级切片：文件头部（imports + docstring）作为 Chunk
└─ 配置文件切片：特殊处理
    ↓
[Embedding] → 向量数据库
    ↓
[元数据索引] ← 文件路径、函数签名、调用关系
```

**为什么不能按字符切？**
> 因为代码的语义单元是函数和类，不是固定字符数。按字符切的 RAG 在代码场景下召回质量很差——你搜"认证逻辑"，它可能给你返回半个函数。

#### 2.3 自然语言问答能力

用户可以对选中项目自由提问：

**基础问题：**
- "这个项目的入口文件在哪里？"
- "主要有哪些模块？它们之间的关系是什么？"
- "用了哪些第三方依赖？"

**进阶问题：**
- "帮我找所有处理用户认证的代码"
- "这个函数做了什么？有没有潜在 Bug？"
- "如果我要加一个 XX 功能，应该改哪些文件？"

**架构问题：**
- "画出这个项目的架构图"
- "数据流是怎么走的？从请求进来到最后响应"
- "哪些模块耦合度较高？"

**实现要求：**
- 回答必须**引用具体代码片段**（文件路径 + 行号范围）
- 支持多轮追问（上下文保持）
- 对于不确定的问题要诚实说"我不确定"，不要瞎编（减少幻觉）

#### 2.4 项目报告生成

最终输出一份**Markdown 格式的项目分析报告**，包含：

```markdown
# [项目名] 深度分析报告

## 1. 基本信息
- 作者、Star 数、许可证、最后更新时间...

## 2. 技术栈分析
- 语言分布、框架版本、核心依赖...

## 3. 架构概览
- Mermaid 架构图（自动生成）

## 4. 模块说明
- 各模块职责、关键类/函数说明...

## 5. 代码质量观察
- 代码风格一致性、测试覆盖率、文档完整性...

## 6. 学习建议（如果是学习用途）
- 建议阅读顺序、重点关注的文件...
```

## 三、技术方案建议

### 推荐技术栈

| 层次 | 推荐 | 备选 |
|------|------|------|
| **LLM** | OpenAI GPT-4o-mini（性价比高） | DeepSeek-V3、Claude Haiku |
| **Embedding** | text-embedding-3-small | bge-m3（开源免费） |
| **向量数据库** | ChromaDB（嵌入式，无需服务器） | Faiss、Qdrant |
| **AST 解析** | tree-sitter（支持多语言） | 内置 ast 模块（仅 Python） |
| **GitHub API** | PyGithub | 直接 requests 调用 |
| **Web 界面** | Streamlit（最快出效果） | Gradio / FastAPI + Vue |
| **状态管理** | SQLite 存储会话和历史 | JSON 文件（简单场景） |

### 为什么推荐这些？

> 学生没有 K8s、没有 GPU 集群、没有付费向量数据库服务。
> 以上所有组件都可以**本地运行**，ChromaDB 是嵌入式的不需要单独部署，
> DeepSeek/bge-m3 可以完全离线跑。降低门槛 ≠ 降低含金量。

## 四、工程化要求（区分课程大作业的关键）

### 必须做到 ✅

| 要求 | 说明 |
|------|------|
| **配置管理** | API Key 通过 `.env` 管理，不硬编码 |
| **错误处理** | GitHub API 限流、网络异常、LLM 调用失败都有优雅降级 |
| **日志系统** | 使用 logging 模块，分级输出 |
| **单元测试** | 核心模块（切片器、RAG Pipeline）必须有测试，覆盖率 > 60% |
| **README 完善** | 安装步骤、使用方式、截图、已知限制 |

### 加分项 ⭐

| 要求 | 说明 |
|------|------|
| **缓存机制** | 搜索结果和分析结果缓存，避免重复调用 API/LLM |
| **流式输出** | 分析过程实时展示进度（不是等很久然后一次性输出） |
| **成本追踪** | 记录每次对话消耗的 Token 和估算费用 |
| **对比实验** | 测试不同的切片策略/Embedding 模型，记录效果差异 |
| **Docker 部署** | 提供 Dockerfile，一键运行 |

## 六、常见坑点提醒（提前告诉候选人）

> 这些是真实开发中会遇到的问题，提前知道比踩坑后补救更有价值

1. **GitHub API 速率限制**：未认证 60 次/小时，认证后 5000 次/小时。要做好缓存和限流处理
2. **代码切片的边界问题**：函数被拆成两半怎么办？跨文件的类怎么处理？
3. **LLM 幻觉**：Agent 可能编造不存在的函数名或文件路径——要有验证机制
4. **大仓库的处理**：有些仓库几万行代码，全部 Embedding 成本很高——要有大小限制策略
5. **Token 上下文窗口**：一次问答可能超出模型上下文长度——要有截断/摘要策略

## 七、进阶挑战（给有余力的候选人）

完成以上内容后，可以继续挑战：

1. **跨仓库对比分析**：同时分析两个类似项目，输出对比报告（优劣势、适用场景）
2. **代码异味检测**：自动识别长函数、深层嵌套、重复代码等问题
3. **贡献热点分析**：基于 Git 历史，找出项目中改动最频繁的文件（暗示复杂度高）
4. **自然语言生成代码修改建议**："给这个函数加上参数校验" → Agent 直接给出 diff
